一步一步用unsloth LoRA微调Qwen3

2026年05月18日 · 12 分钟 · 2406 字 · 八戒

LoRA（Low-Rank Adaptation，低秩微调）。

全面微调（Full Fine-Tuning）一个几百亿参数的大模型像是在“重新装修整栋摩天大楼”（成本极高、极易塌房），那么 LoRA 就是在摩天大楼外面搭几根轻量级的“外挂管道”。

我们来一步一步实现这个过程：

准备#

uv pip install sentencepiece protobuf datasets huggingface_hub hf_transfer

一、加载底模：#

from unsloth import FastLanguageModel
import torch

MODEL = "unsloth/Qwen3-14B"
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name =MODEL,
    max_seq_length = 2048,
    dtype = None,
    load_in_4bit = True,
    full_finetuning = False
)

model = FastLanguageModel.get_peft_model(
    model,
    r = 32,         
    target_modules = ["q_proj", "k_proj", "v_proj", "o_proj",
                      "gate_proj", "up_proj", "down_proj",],
    lora_alpha = 32,
    lora_dropout = 0,
    bias = "none",   
    use_gradient_checkpointing = "unsloth",
    use_rslora = False,  
    loftq_config = None,
)

140 亿（14B 参数） 的庞然大物（通义千问最新版）塞进你的电脑里。

算笔账： 如果按标准出厂格式（Float16，每个参数占 2 字节），140亿参数光是安静地躺在显存里，就需要 28 GB 的显存！如果强行全参微调，显存直接飙升到 80GB 以上。普通人根本玩不起。

max_seq_length = 2048

是模型的“短期记忆草稿纸”大小（上下文窗口）。

为什么不设成 128K？ 虽然 Qwen 支持极长的上下文，但在微调时，长度越长，计算 Attention 时的中间矩阵（特别是 KV Cache）就越庞大。设定为 2048，是为了在“能看懂足够长的问题”和“不把显卡内存撑爆”之间找一个黄金平衡。

full_finetuning = False
load_in_4bit = True

为你下一步接入上一章讲的 LoRA 外挂管道 做准备！

这就是在向 PyTorch 的内存管家下达死命令：“给我把这 140 亿个被压缩成 4-bit 的原生脑细胞彻底冻结（Freeze）！不管等会儿训练老师（Loss 交叉熵）发多大火，绝对不准修改它们中的任何一个数字！”

先用 load_in_4bit = True 把一栋 140 亿块砖头的摩天大楼，强行压缩成了极其轻量的“马赛克微缩模型”，塞进了你的游戏显卡里。

然后你用 full_finetuning = False 给这栋大楼贴上了封条，严禁拆卸。

大模型并不是靠某几个极其精确的数字来思考的，而是靠几百亿个数字的整体网络结构和相对大小。所以即使把精度砍到 4-bit，它的智商也几乎只有极其微小的下降。

二、LoRA微调#

# 获取高效微调模型 (PEFT model)
model = FastLanguageModel.get_peft_model(
    model,
    # 目标模块 (需进行 LoRA 适配的层)
    target_modules = ["q_proj", "k_proj", "v_proj", "o_proj"],
    # 使用梯度检查点 (优化显存技术)，这里指定 Unsloth 的实现
    use_gradient_checkpointing = "unsloth",
    r = 16,            # 秩 (Rank)，低秩适应层的大小
    lora_alpha = 4,    # LoRA 的缩放因子
    lora_dropout = 0,  # LoRA 层的 Dropout 概率 (防止过拟合)
    bias = "none"      # 是否微调偏置 (bias) 项
)

代码使用了目前极其火爆的极速微调库 Unsloth（结合了 HuggingFace 的 PEFT）。每一行都在精妙地控制我们刚才说的那根“外挂管道”：

  • target_modules = ["q_proj", "k_proj", "v_proj", "o_proj"]
    • 含义： 我们要把外挂管道接在哪里？
    • 直觉： 大模型里有无数个矩阵，这里指定只把 LoRA 挂在注意力机制（Attention）的 Q、K、V 和输出层 O 上。这是业界公认性价比最高的“挂载点”。
  • use_gradient_checkpointing = "unsloth"
    • 含义： 开启梯度检查点（一种省显存的黑科技）。
    • 直觉： “时间换空间”。不把所有中间计算结果都存在显卡里，而是用的时候再重新算一遍。unsloth 提供了极其优化的版本，能让你在小显存游戏显卡（比如 RTX 3090/4090）上跑起庞然大物。
  • r = 16 (最关键的参数)
    • 含义： 就是流程图里的那个“秩（Rank）”维度。
    • 直觉： 也就是那个“沙漏”最窄的地方有多宽。$r$ 越大，这根管道能流过去的信息越多，模型能学到的新知识越复杂，但需要的显存也越大。通常设置在 8、16、32 左右。
  • lora_alpha = 4
    • 含义： LoRA 的缩放比例（Scaling factor）。
    • 直觉： 相当于“外挂管道”的音量旋钮。由于最后两路信号要相加（$\oplus$），alpha 决定了外挂信号的声音有多大。在数学上，外挂权重会乘以 $\frac{alpha}{r}$。
  • lora_dropout = 0 和 bias = "none"
    • 含义： 关闭 Dropout，不训练偏置项。
    • 直觉： 极致的抠门学！因为 LoRA 模块本来就极小（参数很少），不容易发生过拟合，所以不需要用 Dropout 来“毒死”神经元。不训练 bias 也是为了进一步榨干显存，把所有算力全用在核心矩阵上。

这张图解释了“如何在不改变原有大模型智商的前提下，教它新知识”。

  1. 左侧的主路（预训练模型）：
    • 这是大模型出厂时自带的“原生知识矩阵”。
    • 注意旁边的字：（微调时冻结）。这意味着在训练时，我们绝对不修改大模型原来的参数（彻底避免了“灾难性遗忘”，不用担心它学了写诗就忘了算数学）。
  2. 右侧的旁路（降维矩阵A 和 升维矩阵B）：
    • 这是我们加装的“外挂管道”（LoRA 模块）。旁边写着（微调时只学习它们）。
    • 核心魔法在于这个“沙漏形”结构：
      • 假设输入数据 x 是 4096 维的。
      • 矩阵 A (降维)： 把 4096 维的信息，狠狠压缩到一个极小的维度 r（比如图中的 $r=16）。
      • 矩阵 B (升维)： 拿着这 16 维的精华特征，再把它重新放大回 4096 维。
  3. 顶部的相加（+）：
    • 原始大模型的输出，和我们“外挂管道”算出来的输出，在最后一步直接相加。
    • 物理直觉： “原生大脑”负责提供基础常识，“外挂大脑”负责提供特定领域的微调风格，两者混合，输出最终结果。

# 样例特征与翻译要求的呼应：
#    思维链 (Chain-of-Thought)： 一开始就是 <think> 标签，里面装满了诸如 "Okay, let's see...", "Let me try that..." 这样的碎碎念。这就是在教模型“三思而后行”，不要急于给出答案，把中间的推理步骤显式地打印出来。
#    清晰标记最终答案 (Mark the final answer clearly)： 在回复的最后，使用了 \boxed{14}。
#    兼容评估基准 (Compatibility with evaluation benchmarks)： 为什么要用 \boxed{}？因为主流的开源数学能力测试集（如 GSM8K, MATH），它们的自动判分脚本都是靠正则表达式去抓取 \boxed{} 里面的内容来判断对错的。如果你不教模型输出这个框，就算它算对了，在测试跑分时也会得零分。    
def generate_conversation(examples):
    problems  = examples["problem"]
    solutions = examples["generated_solution"]
    conversations = []
    for problem, solution in zip(problems, solutions):
        conversations.append([
            {"role" : "user",      "content" : problem},
            {"role" : "assistant", "content" : solution},
        ])
    return { "conversations": conversations, }

reasoning_conversations = tokenizer.apply_chat_template(
    reasoning_dataset.map(generate_conversation, batched = True)["conversations"],
    tokenize = False,
)

Boolean operators are logical operators used in programming to manipulate boolean values. They operate on one or more boolean operands and return a boolean result. The three main boolean operators are "AND" (&&), "OR" (||), and "NOT" (!).

The "AND" operator returns true if both of its operands are true, and false otherwise. For example:

```python
x = 5
y = 10
result = (x > 0) and (y < 20)  # This expression evaluates to True
```

The "OR" operator returns true if at least one of its operands is true, and false otherwise. For example:

```python
x = 5
y = 10
result = (x > 0) or (y < 20)  # This expression evaluates to True
```

The "NOT" operator negates the boolean value of its operand. It returns true if the operand is false, and false if the operand is true. For example:

```python
x = 5
result = not (x > 10)  # This expression evaluates to True
```

Operator precedence refers to the order in which operators are evaluated in an expression. It ensures that expressions are evaluated correctly. In most programming languages, logical AND has higher precedence than logical OR. For example:

```python
result = True or False and False  # This expression is evaluated as (True or (False and False)), which is True
```

Short-circuit evaluation is a behavior where the second operand of a logical operator is not evaluated if the result can be determined based on the value of the first operand. In short-circuit evaluation, if the first operand of an "AND" operator is false, the second operand is not evaluated because the result will always be false. Similarly, if the first operand of an "OR" operator is true, the second operand is not evaluated because the result will always be true.

In programming languages that support short-circuit evaluation natively, you can use it to improve performance or avoid errors. For example:

```python
if x != 0 and (y / x) > 10:
    # Perform some operation
```

In languages without native short-circuit evaluation, you can implement your own logic to achieve the same behavior. Here's an example in pseudocode:

```
if x != 0 {
    if (y / x) > 10 {
        // Perform some operation
    }
}
```

Truthiness and falsiness refer to how non-boolean values are evaluated in boolean contexts. In many programming languages, non-zero numbers and non-empty strings are considered truthy, while zero, empty strings, and null/None values are considered falsy.

When evaluating boolean expressions, truthiness and falsiness come into play. For example:

```python
x = 5
```

To handle cases where truthiness and falsiness are implemented differently across programming languages, you can explicitly check the desired condition. For example:

```python
x = 5
result = bool(x)  # Explicitly converting x to a boolean value
```

        max_steps = 60,

1. 显存刺客的克星：积少成多的“吞咽法”#

  • per_device_train_batch_size = 2 (单次处理量)
    • 物理直觉： 每次只给模型看 2 道题。为什么这么小？因为即使模型被压到了 4-bit，训练时的中间变量（激活值）依然极其庞大。设为 2 是为了保证绝不爆显存（OOM）。
  • gradient_accumulation_steps = 4 (梯度累加步数)
    • 物理直觉： 这是深度学习最伟大的工程发明之一！如果每次只看 2 道题就去更新参数，模型学到的知识会非常“偏科”和“震荡”。
    • 魔法效果： 它告诉模型：“你看完 2 道题，先别急着改参数，把心得体会（梯度）记在脑子里；接着再看 2 道，再看 2 道……直到连续看了 4 批（总共 2 * 4 = 8 道题）之后，把心得体会加在一起，统一修改一次参数！”
    • 结果： 你用极其微小的显存，完美模拟了 Batch Size = 8 的平滑训练效果。

2. 学习节奏的艺术：先热身，再冲刺#

  • learning_rate = 2e-4 (学习率：0.0002)
    • 物理直觉： 决定了模型每次修改参数时的“步子跨多大”。对于 LoRA 微调来说，2e-4 是一个极其经典的“甜点值”。步子太大容易忘记常识（灾难性遗忘），步子太小半天学不会。
  • warmup_steps = 5 (热身步数)
    • 物理直觉： 刚开始训练时，外挂管道里的参数是随机初始化的。如果一开始就用全速（2e-4）去冲，会引发梯度爆炸，瞬间摧毁模型。
    • 魔法效果： 在前 5 步，学习率会从 0 极其平滑地慢慢上升到 2e-4。就像开车起步先轻踩油门。
  • lr_scheduler_type = "linear" (学习率调度器)
    • 物理直觉： 过了 5 步热身期后，随着训练步数越来越接近尾声，学习率会变成直线下降，越来越小。这意味着模型在训练后期会进行更精细的“微调雕刻”，而不是大刀阔斧地乱改。

3. 终极算力压榨：8-bit 优化器#

  • optim = "adamw_8bit"
    • 物理直觉： 训练模型不光需要保存模型本身的参数，还要保存一个叫“优化器状态（动量）”的东西。标准的 32 位 AdamW 优化器是个内存巨兽，会吃掉极大量的显存。
    • 魔法效果： 这是 bitsandbytes 库的又一神作。它把优化器的状态强行压缩成了 8-bit！这能让你额外省出好几个 G 的显存，是单卡微调必开的终极选项。

4. 其他常规设置#

  • max_steps = 60 (总训练步数): 你只打算让模型更新 60 次参数。这通常不是为了训练出一个最终成品，而是一个 “跑通验证 (Sanity Check)” —— 用来测试代码会不会报错、Loss（误差）会不会正常下降。真正训练可能需要几百上千步（通常用 num_train_epochs 控制）。
  • weight_decay = 0.01 (权重衰减): 防止模型死记硬背训练集（过拟合）的惩罚机制。
  • seed = 3407: 固定随机种子，保证每次跑的结果完全一样。（圈内冷知识：3407 是 AI 学术界著名的玄学种子，有一篇著名论文叫《Torch.manual_seed(3407) is all you need》，发现用这个数字通常能跑出好结果。）
  • report_to = "none": 关闭监控面板（比如 WandB），专注于在终端看日志。
